== How the Agents Work

=== The 5-Phase Development Workflow

==== Phase 1: Design (Day 1)

**Architect Agent leads**

Responsibilities::
- Parse requirements and user stories
- Create architecture diagram
- Define domain boundaries and models
- Design API contracts (request/response)
- Identify integration points and dependencies
- Create implementation strategy
- Assess risks and dependencies

Deliverables::
- DESIGN.md with complete specification
- Architecture diagrams (ASCII or PlantUML)
- Domain models and value objects
- API contracts documented
- Implementation strategy
- Risk assessment

Effort:: 1-2 days

[source, markdown]
----
# Feature: Order Service

## Architecture
[ASCII or PlantUML diagram]

## Domain Models
- OrderAggregate: manages order lifecycle
- OrderItem: line item value object
- OrderStatus: state enum

## API Contracts
POST /api/v1/orders
- Request: CreateOrderRequest
- Response: OrderDto

## Dependencies
- IOrderRepository
- IMediator
- ILogger
----

==== Phase 2: Parallel Development (Days 2-3)

Three agents work simultaneously on separate concerns:

===== Implementation Agent

Codes in parallel with testing::
- Domain layer aggregates and value objects
- Application layer handlers and validators
- Infrastructure layer repositories and EF mappings
- API layer endpoints and contracts

Patterns used::
- Domain-Driven Design (DDD)
- Clean Architecture
- CQRS pattern
- Repository pattern
- Async/await throughout

Output::
- Domain: `OrderAggregate.cs`, `OrderItem.cs`, events
- Application: `CreateOrderHandler.cs`, `CreateOrderValidator.cs`
- Infrastructure: `OrderRepository.cs`, `OrderConfiguration.cs`
- API: `OrdersEndpoints.cs`

===== Testing Agent

Writes tests in parallel, using mocks until code ready::
- Unit test fixtures and builders
- Integration test infrastructure with TestContainers
- E2E test scenarios
- Test data factories

Test coverage targets::
- Unit tests: 70% of all tests (high coverage)
- Integration tests: 25% of all tests (critical paths)
- E2E tests: 5% of all tests (user journeys)
- Overall target: >80% code coverage

Output::
- `OrderServiceTests.cs` - Unit tests
- `OrderHandlerIntegrationTests.cs` - Integration tests
- `OrderE2ETests.cs` - End-to-end tests
- `OrderTestDataBuilder.cs` - Test fixtures

===== DevOps Agent

Prepares infrastructure and deployment in parallel::
- GitHub Actions workflow templates
- AWS CDK infrastructure as code
- Environment configuration (dev, staging, prod)
- Logging and monitoring setup
- Health checks and observability

Output::
- `.github/workflows/deploy.yml` - CI/CD pipeline
- `infra/order-service-stack.ts` - AWS CDK
- Kubernetes manifests (if applicable)
- Monitoring and alerting configuration

==== Phase 3: Integration (Day 4)

**Three agents integrate their work**

Testing Agent::
- Finalizes integration tests with real code
- Achieves coverage targets
- Validates performance
- Creates test report

Review Agent::
- Begins static code analysis
- Security scanning
- Performance review
- Architecture validation

DevOps Agent::
- Deploys to staging environment
- Configures databases
- Sets up monitoring
- Validates infrastructure

==== Phase 4: Quality Assurance (Days 4-5)

**Review Agent leads quality gate**

Validation checklist::
- âœ… Code compiles without warnings
- âœ… All tests pass (>80% coverage)
- âœ… Security scan passes (zero critical issues)
- âœ… Performance meets SLA
- âœ… Architecture compliant
- âœ… Documentation complete
- âœ… Staging deployment successful

Decision::
- **Approve**: Ready for merge and production
- **Conditional**: Address feedback and re-review
- **Reject**: Fix critical issues and resubmit

==== Phase 5: Deployment (Day 5+)

**DevOps Agent leads deployment**

Sequence::
1. Merge to main branch (Review approval)
2. CI/CD pipeline triggers automatically
3. Build and test (parallel)
4. Security scan
5. Docker image creation and push
6. Deploy to staging
7. Smoke tests validation
8. Deploy to production (blue-green)
9. Monitoring and validation
10. Rollback available if needed

Time estimate:: 2-8 hours

namespace Avro.Orders.Domain;

// Record for immutable value objects
public record OrderItem(string ProductId, int Quantity, decimal Price);

// Aggregate root with business logic
public class OrderAggregate : AggregateRoot
{
    private List<OrderItem> _items = new();
    
    // Factory method
    public static OrderAggregate Create(string tenant, string customerId)
    {
        var order = new OrderAggregate 
        { 
            Id = Guid.NewGuid().ToString(),
            Tenant = tenant,
            CustomerId = customerId,
            Status = OrderStatus.Draft
        };
        
        order.AddDomainEvent(new OrderCreatedEvent(order.Id, customerId));
        return order;
    }
    
    // Business operation
    public void Confirm(string confirmedBy)
    {
        if (Status != OrderStatus.Draft)
            throw new InvalidOperationException("Order must be in Draft status");
            
        Status = OrderStatus.Confirmed;
        AddDomainEvent(new OrderConfirmedEvent(Id, confirmedBy));
    }
}

== Troubleshooting

=== Agent Blocked?

**If an agent can't proceed**:

1. Check status in task.md
2. Identify blocker
3. Document in GitHub discussion
4. Get approval from blocking agent
5. Continue with approved changes
6. No work should block entirely

Examples::
- **Implementation blocked on design**: Proceed with approved assumptions, update when design clarified
- **Testing blocked on code**: Write tests with mocks, finalize when code ready
- **DevOps blocked on architecture**: Use manual steps, automate when ready

=== Quality Gate Failing?

**If Review Agent rejects code**:

1. Review ðŸ”´ Critical issues (must fix)
2. Address ðŸŸ¡ Major issues (should fix)
3. Consider ðŸŸ¢ Minor suggestions
4. Get Review Agent sign-off
5. Request approval for merge

=== Timeline Slipping?

**If feature taking >5 days**:

1. Check which agent is bottleneck
2. Add resources or reassign if possible
3. Document learnings
4. Adjust for next feature
